# 半天速成精读表

这份文档适合想在短时间内快速建立项目全局理解的同学。

目标不是逐行背代码，而是在 4 到 5 小时内回答清楚下面 5 个问题：

1. `/api/chat` 请求是怎么进系统、怎么返回的。
2. `Supervisor` 为什么存在，它怎么编排其他 Agent。
3. 这个项目里的 RAG 流程具体做了什么。
4. 长短期记忆分别负责什么。
5. MCP 工具层和聊天链路目前是什么关系。

## 使用方法

每看完一个阶段，都做 3 件事：

- 用一句话写下这个文件在系统里的角色。
- 画出它和上下游文件的调用关系。
- 至少做一次本地验证，不要只看不跑。

## 阶段 0：5 分钟建立地图

先看：

- [README.md](../README.md)

这一阶段只抓 4 个信息：

- 这是一个多 Agent 智能客服项目。
- 入口是 FastAPI。
- 核心编排是 `Supervisor + LangGraph`。
- 能力大致分成 RAG、工单、合规、工具调用。

看完后你应该能说：

- 这个项目不是单个聊天机器人，而是多个 Agent 分工协作。

## 阶段 1：30 分钟看系统入口

先看：

- [api/main.py](../api/main.py)

重点看这些位置：

- 应用初始化：`WorkingMemory`、`ShortTermMemory`、`LongTermMemory`、`mcp_server`
- `lifespan()`：启动时做了什么
- `/api/chat`
- `/api/tools`
- `/api/tools/call`

这一阶段要搞懂：

- 请求从哪里进入系统。
- `initial_state` 是怎么构造的。
- `graph.ainvoke(...)` 是什么时候被调用的。
- 为什么这个接口会“等整条链路跑完”再返回。

建议验证：

- 访问 `GET /health`
- 访问 `GET /api/tools`
- 用 PowerShell 调一次 `POST /api/chat`

看完后你应该能说：

- `main.py` 是系统入口，负责启动应用、注册 API、初始化记忆和工具，并把聊天请求交给 `LangGraph` 图执行。

## 阶段 2：50 分钟看编排核心

先看：

- [agents/supervisor.py](../agents/supervisor.py)

重点看这些函数：

- `route_decision()`
- `synthesize_response()`
- `create_supervisor_graph()`
- `route_to_agent()`

这一阶段要搞懂：

- 为什么需要 `Supervisor`，而不是直接让一个 Agent 回答。
- 状态 `AgentState` 里都存了什么。
- 路由是怎么决定走 `knowledge_rag`、`ticket_handler` 或 `compliance_check` 的。
- 为什么最后还要再经过一次 `synthesize_response()`。

建议你边看边画图：

- `supervisor_route -> knowledge_rag/ticket_handler -> compliance_check -> synthesize`

建议验证：

- 对照 `/api/chat` 的 `initial_state` 看 `AgentState` 如何流动。

看完后你应该能说：

- `supervisor.py` 是全项目核心，负责路由决策、图编排和最终结果汇总。

## 阶段 3：45 分钟看 RAG 主流程

先看：

- [agents/knowledge_rag.py](../agents/knowledge_rag.py)

重点看这些函数：

- `rewrite_query()`
- `retrieve_documents()`
- `rerank_documents()`
- `generate_answer()`
- `process()`

这一阶段要搞懂：

- 这个项目的 RAG 不是一句话，而是 4 步串起来的。
- 为什么先改写 query，再检索，再重排，最后生成回答。
- `process()` 最后写回到了哪里。

建议你记住这条链：

- Query Rewrite
- Retrieve
- Rerank
- Generate

建议验证：

- 找出“知识库没找到”这句回复是在哪里返回的。

看完后你应该能说：

- `knowledge_rag.py` 实现了一个简化但完整的 RAG 流程，并把结果写回 `sub_results["knowledge_rag"]`。

## 阶段 4：35 分钟看长期记忆和向量检索

先看：

- [memory/long_term.py](../memory/long_term.py)

重点看这些函数：

- `_init_index()`
- `_simple_embedding()`
- `add_document()`
- `search()`
- `load_knowledge_base()`

这一阶段要搞懂：

- 为什么它叫“长期记忆”。
- 它和 RAG 的关系是什么。
- FAISS 在这里起什么作用。
- 为什么当前实现更像演示版，而不是生产版。

这一步尤其要注意：

- `_simple_embedding()` 不是正式 embedding 服务，而是一个简化实现。

建议验证：

- 对照 `api/main.py` 看启动时哪些知识被写入了 `long_term_memory`。

看完后你应该能说：

- `LongTermMemory` 是 RAG 的检索底座，但当前 embedding 方案是简化版，适合学习，不适合直接当生产实现。

## 阶段 5：25 分钟看短期记忆

先看：

- [memory/short_term.py](../memory/short_term.py)

重点看这些函数：

- `_get_redis()`
- `add_message()`
- `get_history()`
- `get_context_window()`

这一阶段要搞懂：

- Redis 可用时和不可用时分别怎么工作。
- 多轮对话上下文是怎么保存的。
- 为什么它叫“短期记忆”。

看完后你应该能说：

- `ShortTermMemory` 负责会话级上下文保存，优先用 Redis，不可用时降级到内存。

## 阶段 6：35 分钟看工具层

先看：

- [mcp/mcp_server.py](../mcp/mcp_server.py)

重点看这些位置：

- `MCPToolServer`
- `register()`
- `call_tool()`
- `create_default_tools()`
- `order_query`

这一阶段要搞懂：

- MCP 工具层是什么。
- `order_query`、`ticket_create`、`risk_check` 是怎么注册进去的。
- 工具和聊天主链路为什么不是一回事。

建议验证：

- 调用 `GET /api/tools`
- 调用 `POST /api/tools/call`

推荐请求体：

```json
{
  "name": "order_query",
  "arguments": {
    "user_id": "user_001"
  }
}
```

看完后你应该能说：

- 这个项目里有 `order_query` 工具，但它目前还没有真正接进 `/api/chat` 的聊天链路。

## 阶段 7：30 分钟看合规和业务 Agent

先看：

- [agents/compliance_checker.py](../agents/compliance_checker.py)
- [agents/ticket_handler.py](../agents/ticket_handler.py)

这一阶段不用逐行死抠，重点看职责划分：

- `compliance_checker.py` 代表安全、规则和审查
- `ticket_handler.py` 代表结构化业务处理

这一阶段要搞懂：

- 为什么客服系统里不只有 RAG。
- 为什么最终回答前还要合规检查。
- 为什么工单类请求更像业务动作而不是知识问答。

看完后你应该能说：

- 这个项目不是只有“问答”，还包含业务执行和内容治理两个方向。

## 阶段 8：20 分钟做一次复盘

现在不要再看新文件，直接自己回答下面这些问题：

1. `/api/chat` 收到请求后，依次会经过哪些核心模块。
2. `Supervisor` 在这个系统里真正解决了什么问题。
3. RAG 的 4 个步骤分别是什么。
4. `LongTermMemory` 和 `ShortTermMemory` 的区别是什么。
5. `order_query` 为什么“存在于仓库里”但“还没真正用于聊天”。

如果这 5 个问题你能不用看代码回答出来，速成阶段就完成了。

## 半天后你该进入什么阶段

如果你已经完成这份精读表，下一步最值得做的是二选一：

- 做一次小改动：把“查询订单状态”真正接到 `order_query` 工具上。
- 做一次工程增强：给 `/api/chat` 加超时和更清晰的错误处理。

这两件事都比继续漫无目的读文件更有效，因为它们会把“看懂代码”变成“真的掌握项目”。

## 最终输出建议

学完这半天后，建议你自己整理一页笔记，标题就叫《smart-cs-multi-agent 项目速记》。

至少写下这 6 行：

- 项目定位：多 Agent 智能客服原型
- 入口文件：`api/main.py`
- 编排核心：`agents/supervisor.py`
- RAG 核心：`agents/knowledge_rag.py`
- 检索底座：`memory/long_term.py`
- 工具层：`mcp/mcp_server.py`

只要这 6 行你能自己展开讲，这个项目就不再只是“看过”，而是“已经入门了”。
